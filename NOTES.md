### Уязвимости и проблемы безопасности

| Проблема | Файл | Риск | Решение |
|----------|------|------|---------|
| **Черновики в памяти** | `main.py:55` | 🔴 Высокий | При перезапуске бота все `drafts` теряются |
| **Нет rate limiting** | `main.py` | 🔴 Высокий | Можно спамить команды, DOS бота |
| **Singleton без реконнекта** | `client.py:8` | 🟡 Средний | `_client` не обрабатывает разрыв соединения с Supabase |
| **Слабая аутентификация** | `main.py:47` | 🟡 Средний | Проверка только по `chat_id` — легко подделать |

**Примеры проблемного кода:**
```python

# Нет валидации длины входных данных
```

### Сложность кода

#### ✅ Хорошо
- Разделение на агентов — low coupling
- Функции в целом небольшие (<50 строк)
- Чёткие Pydantic-модели

#### ⚠️ Проблемные места

**A. Дублирование кода (DRY нарушен)**
```python
# _get_bot() дублируется в 2 файлах:
# coordinator.py:22 и publisher_agent.py:16
# Решение: вынести в shared/utils.py
```

**B. Сложная inline-логика в обработчиках**
- `callback_post_action` — 67 строк, сложно тестировать

**C. Многоуровневая вложенность в `run_multi`**
Функция делает слишком много:
1. Загружает твиты
2. Фильтрует и ранжирует
3. Группирует по топикам
4. Генерирует посты
5. Проверяет уникальность
6. Маркит твиты как обработанные

**D. Распределённое глобальное состояние**
```
client.py:4   → _client (DB)
main.py:55    → drafts (bot_data)  
scheduler.py   → jobs (JSON-файл)
```

### Масштабируемость

#### 🔴 Блокеры масштабирования

| Проблема | Почему это проблема | Когда проявится |
|----------|---------------------|---------------|
| **In-memory drafts** | Нельзя запустить 2 инстанса бота | При горизонтальном масштабировании |
| **APScheduler** | Не работает в distributed mode | При нескольких серверах |
| **JSON-файлы для конфига** | Нет atomic writes, race conditions | При одновременном редактировании |
| **Нет очереди задач** | Pipeline выполняется синхронно | При росте нагрузки |
| **Нет кэширования** | Каждый запрос к Supabase — сетевой вызов | При частых операциях |

#### 📊 Оценка готовности к масштабированию

| Компонент | Текущий статус | Макс. нагрузка |
|-----------|----------------|----------------|
| Scraper | ✅ Stateless | 100+ тегов |
| Analyzer | ⚠️ Sync блокировки | 5-10 постов/час |
| Publisher | ⚠️ Retry в памяти | 1 канал |
| Scheduler | ❌ Single-instance | 1 сервер |
| Database | ⚠️ Нет пула соединений | ~100 req/sec |
